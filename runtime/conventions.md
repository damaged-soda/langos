# 基础规范（conventions）

本文件定义 AI 参与协作时的基础规范，覆盖命名、落盘与写作要求，直接影响运行时行为，属于“必读最小集”。

## 1. 通用原则
- 先问再写：信息不足先澄清，必要假设要标注并复述。
- 复述确认：涉及修改/写入/执行前，先复述预期变更与影响，得到确认后再行动。
- 留痕审计：关键输入/输出/决策要可追溯，便于复盘。

## 2. 语言与编码
- 对话用中文，代码/命令/路径保持原文。
- 新文件默认使用 ASCII；无明确理由不引入非 ASCII。

## 3. 目录与文件类型（示例）
- 业务文档库由用户指定（如 `workspace/` 或独立 docs 仓），用于放置业务/背景文档，可选但不影响运行时最小集。  
- 协议与运行时：`runtime/`。  
- 本规范：`runtime/conventions.md`。  
- 文档库示例：`../langos-docs/`（高层背景、愿景、自举说明等，供参考，非运行时依赖）。

## 4. 命名规范
- 日期：`YYYYMMDD`，时间 `YYYYMMDD-HHMM`（24 小时制）。
- 文档文件名：小写连字符，如 `feature-x-spec.md`。
- 协议文件名：`<id>.yaml`，id 用小写字母/数字/下划线，语义清晰。

## 5. 文档结构与写作方式
- 层次清晰：标题从 `#` 开始递进；同级用一致格式。
- 段首给结论/要点，必要时用短列表。
- 引用他处文档时给出相对路径。

## 6. AI 行为约定（基础层）
- 不编造：无法确定的信息要说明不确定或询问。
- 提醒确认：写入/执行前再次确认。
- 变更影响：修改协议/规范时说明对现有流程的影响。

## 7. 版本与演进
- 小步修改优先，重大变更先提方案。
- 如变更影响范围大，建议在文档库记录原因（可用 ADR）。

## 8. 实际使用建议
- 修改文档或协议时，同步更新索引（如 `runtime/protocols/index.yaml` 或 docs 仓索引）。
- 如文件过多，按子目录或主题拆分，保持可浏览性。
